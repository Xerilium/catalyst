# Feedback for workflow-context

## Distilled-writing

- Distilled-writing (and other content-quality constraints) currently get scattered as per-workflow FRs (e.g., `FR:workflow.distilled-writing` in feature-workflow). This duplicates across every workflow and creates drift risk. The right scope is global, but how do we enforce globally without maintaining a separate registry?
  - **Constraint**: Don't maintain a list of files that need distillation separately from the files themselves. Single source of truth.
  - **Option A — Opt-out by default (simplest)**: Assume EVERY action playbook and spec template must follow Distilled Excellence. Maintain a small opt-out list inline in the standard itself for files where verbosity is intentional (e.g., onboarding docs, examples). Pros: zero metadata to maintain in normal case; new files automatically covered. Cons: opt-out list still exists, just smaller.
  - **Option B — Marker in the file**: Templates and action playbooks that target AI-context-generation include a frontmatter or comment marker (e.g., `distilled: true`) that the validator picks up. Pros: discoverable by reading the file; no separate list. Cons: requires every file to declare; risk of forgetting on new files (defaults to non-enforced unless we flip the default).
  - **Option C — Path convention**: Files in specific paths (`src/resources/playbooks/`, `src/resources/templates/specs/`) are automatically in scope. Pros: no per-file metadata; new files in those paths automatically covered. Cons: hard-codes path conventions; doesn't extend to AI-context files outside those paths.
  - **Recommendation**: Option A (opt-out by default) is simplest and aligns with how spec templates already work — every spec template needs to be distilled, so we're effectively already there. The opt-out list lives in the Distilled Excellence standard file (or wherever the rule is defined) so it stays adjacent to the rule.
  - **Related**: A more comprehensive Distilled Excellence experience is tracked in feature-context feedback. If that work moves into workflow-context (or its own feature), this enforcement decision should land there.
  - **How to apply**: When the broader Distilled Excellence work is picked up, decide A/B/C and remove the per-workflow `FR:workflow.distilled-writing` duplicates from feature-workflow and any other workflow specs in favor of the global enforcement mechanism.
- "Distillery" Claude plugin idea — refined, goal-oriented version of the Caveman plugin built around the **Distilled Excellence** engineering principle. Caveman drops articles and transitions to force terseness; Distillery would enforce the same outcome (cut filler, hedges, restatement) without sacrificing quality (keep words that improve outcomes). Worth building if the in-repo principle proves effective.
  - **How to apply:** Before creating an AUQ, ask: "Is there genuine ambiguity here, or am I just seeking permission?" If the user's intent is clear and only one option makes sense, execute directly. Reserve AUQ for real choices with tradeoffs.

## Shared workflow actions

- `feature-state.md` (the Active State update ritual at every STOP gate) is invoked by every orchestration playbook (create-feature.md, start-blueprint.md, future workflows). It's a workflow-level convention, not a feature-workflow-specific concern, but it currently lives at `src/resources/playbooks/actions/feature-state.md` (the `feature-` prefix implies feature-workflow ownership).
  - **Why**: Same drift-risk pattern as auq-usage / distilled-writing. Every workflow spec needs to declare a dependency on this ritual; right now blueprint-workflow spec has `FR:workflow.continuity` with `@req` to `FR:feature-context/rollout.active-state` (the spec, not the action). The action itself is owned ambiguously.
  - **How to apply**: Rename `feature-state.md` → `workflow-state.md` (or `update-state.md`) when convenient. Update all callers (create-feature.md, start-blueprint.md, future workflows) and tests that reference it. Optional: move `actions/` ownership from feature-workflow to workflow-context conceptually (the file path doesn't need to change). Workflow specs should `@req` the workflow-context FR that declares the ritual is required across all workflows — that FR doesn't exist yet either, so add it to workflow-context spec when this work happens.
- `feature-scope.md` § Step 1 item 9 (resume-from-rollout routing) hardcodes per-phase artifact assumptions: "does `.xe/features/{feature-id}/spec.md` exist", "are tests written", "does code exist that would make tests pass". These don't apply to non-feature workflows (blueprint-workflow has no per-feature spec, no tests, no code). The phase-level routing still works, but the artifact-existence checks are wrong for blueprint and any future non-feature workflow.
  - **Why**: blueprint-workflow has 4 phases (Scope → Plan → Implement → Review) with different artifact shapes (one blueprint.md instead of spec.md + tests + code). Resuming a blueprint rollout shouldn't ask "does spec.md exist" — it should ask "does blueprint.md exist."
  - **How to apply**: Generalize feature-scope.md item 9 to phase-level routing without artifact-name hardcoding, OR extract the resume logic into its own action that each workflow can wire to its own artifact-existence checks. Lower priority — the current logic is permissive enough that blueprint resume works in practice; it just asks slightly nonsensical questions.
- Consider renaming `feature-complete.md` → `workflow-review.md` (or extracting a shared base action) to reflect that it works for both feature and blueprint workflows.
  - **Status**: Partially addressed — feature-complete.md amended with conditional language for blueprint workflows (spec-recovery action chain, PR title type now allows `Blueprint`). Still named `feature-*` which conceptually misrepresents its scope.
  - **Why**: The action is now generic. The `feature-` prefix implies feature-workflow ownership. Same shape as the feature-state.md naming concern flagged in workflow-context feedback.
  - **How to apply**: When the workflow-context constraint consolidation lands, rename feature-state.md and feature-complete.md (and any other shared actions) to drop the workflow-specific prefix. Update all callers. Lower priority — current naming works, just misleading.
